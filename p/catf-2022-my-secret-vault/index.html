Un ptrUserInput à seulement 3 caractères 🤔 ? Il s’avère que le pseudo-décompilateur n’a pas su quelle était la taille du tableau de caractères (string) . Une des premières conditions vérifiées est la suivante :

La taille de l’entrée utilisateur doit être supérieur ou égal à 25 pour ne pas rentrer dans la condition (évitant le exit(0)) et est donc de 26 caractères minimums (strlen ne prend pas en compte le NULL BYTE à la fin donc 25+1).

III. Résolution du système

Pour résoudre toutes ces contraintes sur l’entrée utilisateur que l’on a vue plus haut, il y a trois possibilités :

Bien que je l’aie initialement résolu à la main, nous allons ici utiliser Z3, quant à angr, l’épave qui me sert de PC est incapable de trouver en temps raisonable un chemin vers la victoire (donc si certains ont un script fonctionnel je suis preneur).

Z3

Pour résumer très brièvement, z3 (https://github.com/Z3Prover/z3) permet, avec un certains nombre de contraintes/formules donné, de déterminer s’il existe une solution à ce problème, s’il est satisfaisable.

pip install z3-solver

Bon pour des contraintes aussi simples, c’est un peu sortir l’arme nucléaire pour tuer une mouche mais c’est un bon cas d’école pour découvrir z3. Sans plus tarder voilà le script après avoir copié/collé la majorité des contraintes depuis IDA :

import z3

# Prepare
userInput = z3.IntVector("userInput", 26)
s = z3.Solver()

# Constraints
s.add(userInput[5] == 45, userInput[11] == 45, userInput[18] == 45)
s.add(
    userInput[0] == userInput[10],
    userInput[1] == 101,
    userInput[3] == 122,
    userInput[25] == 0,
    userInput[2] == 101,
)
s.add(
    userInput[4] == userInput[17] + 2,
    userInput[6] == 120,
    userInput[7] == 100,
    userInput[8] == userInput[22],
    userInput[9] == 76,
)
s.add(
    (2 * userInput[5] + 9) == userInput[12],
    userInput[23] == userInput[17] + 1,
    userInput[13] == 116,
    userInput[14] == 102,
)
s.add(
    2 * userInput[15] - 134 == userInput[16],
    userInput[21] == 84,
    userInput[16] == 72,
    userInput[20] == 117,
    userInput[17] == 53,
)
s.add(
    userInput[19] == 112,
    userInput[22] == 70,
    userInput[10] == userInput[21],
    2 * userInput[24] - 61 == userInput[20],
)

# Solve & Print result
if s.check():
    m = s.model()
    solution = "".join([chr(m.eval(userInput[i]).as_long()) for i in range(26)])
    print(
        "https://cyberavent.ctfd.io/files/21c81d37fdb40d831bc68ca74cbf2084/" + solution
    )

Il s’agit cette fois-ci d’un binaire pour un OS Windows.

La commande strings va nous donner une information essentielle qui explique je pense pourquoi beaucoup de gens n’ont pas réussi à passer cette étape :

_CorExeMain est une caractéristique propre au framework .NET (ce qui explique pourquoi Ghidra et IDA Freeware sont en difficulté pour désassembler).

On va donc devoir utiliser un outil adapté au reverse de binaire issu du framework .NET (qui est généralement du C#) le plus connu de tous étant :

  • ILSpy (https://github.com/icsharpcode/AvaloniaILSpy) pour OS Linux

Une fois importé, on a le main suivant :

Le binaire n’est pas bien grand et le flag est généré comme suit (après quelques renommages) :

private static double LuysGrAgZzbIRsd() {
	return (double)(DateTime.get_Now().get_Month() * DateTime.get_Now().get_Year());
}

[...]

// La partie mystère
double mysterious = Program.LuysGrAgZzbIRsd(); // ??
int mysteriousDivided = (int)(mysterious / 227.475); // ??/227.475
string strMysteriousDiv = string.Concat("0x", Program.int_to_hex(mysteriousDivided)); // str(hex(??/227.475))
int keyXOR = Convert.ToInt32(strMysteriousDiv, 16); // ??/227.475 (mysteriousDivided==keyXOR)
string str3 = Program.xor_string("31$6+e5&5e#9>7}3s}9#}51#)}", keyXOR);

// La partie affichage
Program.green_text();
string str4 = string.Concat(str3, mysterious.ToString());
string str5 = str4.Insert(5, mysterious.ToString());
string str6 = str5.Insert(10, "-");
string str7 = str6.Replace("catf", "CATF"); // La ligne qui débloque le mystère
string str8 = string.Concat(str7, "}");

Nous n’avons pas connaissance de la variable mysterious qui conditionne la génération du flag, cependant dans la fin du morceau de code on obtient une indication précieuse : après avoir “““deXORé””” la chaîne 31$6+e5&5e#9>7}3s}9#}51#)}, cette dernière semble contenir catf .

On va donc bruteforce la valeur de keyXOR (qui est utilisée pour le XOR) et remonter petit à petit pour retrouvé mysterious qui sera la seule partie manquante du flag. On va pour ça s’aider de CyberChef (https://gchq.github.io/CyberChef/) :

🚨 CyberChef renvoie la clé en hexadécimal 🚨 (temps phénoménal perdu ici)

Key = 50: catf{5eve5sing-c#-is-easy-
  • Ca ressemble bien à un début de flag, donc keyXOR = 0x50 = 80.
  • keyXOR provient de la variable strMysteriousDiv qui était exprimée en hexadécimal, donc strMysteriousDiv = ‘0x50’.
  • strMysteriousDiv s’est formée à partir de mysteriousDivided qui était exprimée en décimal, donc mysteriousDivided = 0x50 = 80.
  • mysteriousDivided est issue de la division de mysterious par 227.475, donc mysterious ≈ 80 * 227.475 ≈ 18198 (notez bien le ≈).

La dernière condition semble finaliser le reverse mais il y a une subtilité :

int mysteriousDivided = (int)(mysterious / 227.475); // Doit être égal à 80

La division par un nombre flottant et le cast en (int) fait perdre de la précision et il n’y a donc pas unicité de la solution pour la valeur mysterious :

mysterious = 18198 => int(18198 / 227.475) = 80 
mysterious = 18199 => int(18199 / 227.475) = 80
mysterious = 18200 => int(18200 / 227.475) = 80

etc ...

Il va donc falloir trouver quelle est réellement la valeur mysterious. En revenant dans le code, on remarque qu’elle est initialisée par la fonction suivante :

private static double LuysGrAgZzbIRsd() {
	return (double)(DateTime.get_Now().get_Month() * DateTime.get_Now().get_Year());
}

mysterious est issu d’un produit du mois et de l’année au moment de l’exécution. Un petit script python pour énumérer les cas possibles :

for i in range(1, 12 + 1):  # Month
    for j in range(2000, 2022 + 1):  # Years
        if int((i * j) / 227.475) == 80:  # Constraint
            print(i, j)

On obtient :

$ python3 solver.py

9 2022

Cela semble très cohérant avec la période du CTF, mysterious = 9 * 2002 = 18198

Il ne reste plus qu’à regarder les dernières lignes pour rassembler le tout et on obtient le flag suivant :

CATF{18198-5eve5sing-c#-is-easy-18198}
